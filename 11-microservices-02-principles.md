# Решение задач по проектированию инфраструктуры

## Задача 1: Обеспечение разработки (CI/CD)

### Предлагаемое решение: GitLab (GitLab CI/CD)

Мы предлагаем использовать платформу **GitLab**, так как она закрывает все требования «из коробки»: от хранения кода до поставки приложений, объединяя в себе как систему контроля версий, так и мощный CI/CD инструмент.

### Состав решения и взаимодействие компонентов

1.  **Хранение кода (Git):**
    *   Используется встроенный в GitLab Git-репозиторий.
    *   **Принцип:** «Один репозиторий на сервис» (Microservice repository pattern). Это позволяет изолировать конфигурации сборки и права доступа для каждой команды.

2.  **Непрерывная интеграция и поставка (CI/CD):**
    *   **Движок:** GitLab Runner (агенты сборки).
    *   **Управление:** Встроенный модуль GitLab CI/CD.

### Схема взаимодействия

1.  Разработчик пушит код в репозиторий GitLab.
2.  GitLab обнаруживает событие `push` и смотрит на наличие файла `.gitlab-ci.yml` в корне репозитория.
3.  GitLab создает задачу (job) и отправляет её в пул свободных **GitLab Runners**.
4.  Runners выполняют шаги сборки, тестирования и деплоя, описанные в YAML-конфигурации.
5.  Артефакты сборки (например, Docker-образы) публикуются в Container Registry (встроенный в GitLab или внешний).
6.  Статус выполнения и логи отображаются в веб-интерфейсе GitLab.

### Соответствие требованиям (Обоснование)

| Требование | Реализация в GitLab |
| :--- | :--- |
| **Облачная система** | GitLab.com — это облачный SaaS. Также можно развернуть Self-Managed в любом облаке (AWS, GCP, Yandex Cloud). |
| **Система контроля версий Git** | Нативная поддержка Git. |
| **Репозиторий на каждый сервис** | Поддерживается созданием отдельных проектов. |
| **Запуск сборки по событию из VCS** | Триггеры по `push`, `merge request`. |
| **Запуск сборки по кнопке с параметрами** | **Manual jobs** с параметрами (`when: manual` + `variables`). Можно запустить пайплайн вручную, указав переменные. |
| **Привязка настроек к каждой сборке** | Переменные окружения, заданные в CI/CD настройках проекта или прямо в пайплайне. |
| **Шаблоны для конфигураций сборок** | Использование `extends` и `!reference` в `.gitlab-ci.yml`, а также включение внешних шаблонов (`include`). |
| **Безопасное хранение секретов** | Встроенное хранилище **CI/CD Variables** (зашифрованы). Интеграция с внешними секретницами: **HashiCorp Vault**, AWS Secrets Manager. |
| **Несколько конфигураций сборки из одного репозитория** | Различные схемы пайплайнов через rules, ветки, теги, или использование `include: local` для разных файлов конфигурации. |
| **Кастомные шаги при сборке** | Любые шаги прописываются в `script:` (bash, powershell, вызов Makefile). |
| **Собственные Docker-образы для сборки** | Использование ключевого слова `image: <my-custom-image>` для каждого job'а. |
| **Развернуть агентов сборки на своих серверах** | Установка **GitLab Runner** на собственные сервера (bare-metal или виртуалки). Регистрация раннера в проекте или группе. |
| **Параллельный запуск нескольких сборок** | Настройка количества `concurrent` заданий в конфигурации раннера. Распределение нагрузки между несколькими раннерами. |
| **Параллельный запуск тестов** | Распараллеливание тестов на уровне CI/CD (матрица стратегий `parallel:matrix`), либо вызов тестового фреймворка в параллельном режиме внутри одного джоба. |

## Задача 2: Логи

### Предлагаемое решение: Стек ELK (Elasticsearch + Logstash + Kibana)

Это отраслевой стандарт для централизованного сбора и анализа логов, обладающий высокой гибкостью и мощным интерфейсом поиска.

### Состав решения и взаимодействие компонентов

1.  **Сбор и транспортировка:** Filebeat (устанавливается на каждом хосте).
2.  **Буферизация и парсинг:** Logstash (или Kafka + Logstash для высоких нагрузок).
3.  **Хранение и индексация:** Elasticsearch.
4.  **Визуализация и UI:** Kibana.

### Схема взаимодействия

1.  **Приложения** пишут логи в **stdout/stderr** (в контейнерах) или в файлы на хосте.
2.  **Filebeat** (агент на каждом узле) читает логи из файлов/контейнерных драйверов.
3.  Filebeat отправляет логи напрямую в **Logstash** (или в Redis/Kafka для надежности).
4.  **Logstash** парсит, обогащает и преобразует логи, после чего отправляет в **Elasticsearch**.
5.  **Elasticsearch** индексирует документы.
6.  Разработчик заходит в **Kibana**, чтобы искать логи, строить дашборды и делиться ссылками на поиск.

### Соответствие требованиям (Обоснование)

| Требование | Реализация |
| :--- | :--- |
| **Сбор в центральное хранилище со всех хостов** | Filebeat ставится на каждый хост (Docker host или VM) и шлет данные в центральный кластер. |
| **Минимальные требования к приложениям / сбор из stdout** | Приложениям достаточно писать в stdout/stderr. Docker или Kubernetes собирает это, Filebeat забирает из лог-файлов контейнера (или использует драйвер контейнера). |
| **Гарантированная доставка** | Filebeat имеет механизмы подтверждения доставки (acknowledgments) и хранит состояние отправленных файлов. При падении Logstash/сети, Filebeat продолжает читать лог с места остановки после восстановления связи. |
| **Поиск и фильтрация** | **Elasticsearch** — поисковый движок, построенный на Apache Lucene. Позволяет делать полнотекстовый поиск, фильтры по полям, range-запросы. |
| **Пользовательский интерфейс для разработчиков** | **Kibana** предоставляет веб-интерфейс (Discover), где разработчики могут искать по логам. Доступ настраивается через роли (RBAC) в Elasticsearch + Kibana Spaces. |
| **Возможность дать ссылку на сохранённый поиск** | В Kibana любой поиск или фильтр можно сохранить как "Saved Search" и поделиться прямой ссылкой на него с коллегами. |

## Задача 3: Мониторинг

### Предлагаемое решение: Prometheus + Grafana

Связка Prometheus и Grafana является стандартом *de facto* для мониторинга динамических сред, таких как микросервисные архитектуры и Kubernetes.

### Состав решения и взаимодействие компонентов

1.  **Сбор метрик (Pull-модель):** Prometheus Server.
2.  **Экспортеры:**
    *   **Node Exporter** — для метрик хостов (CPU, RAM, Disk, Network).
    *   **cAdvisor** (или встроенный в kubelet) — для метрик контейнеров/сервисов (потребление ресурсов).
3.  **Специфичные метрики сервисов:** Клиентские библиотеки Prometheus, встроенные в код приложений (на Go, Java, Python, etc.).
4.  **Визуализация:** Grafana.

### Схема взаимодействия

1.  На каждом хосте работает **Node Exporter** (метрики ОС) и **cAdvisor** (метрики контейнеров).
2.  В каждом микросервисе инструментирован код (/metrics endpoint).
3.  **Prometheus** по расписанию (scrape interval) "ходит" на все известные ему эндпоинты (хосты и сервисы) и забирает метрики (pull-модель).
4.  Prometheus хранит временные ряды данных в своей TSDB (Time Series Database).
5.  **Grafana** подключается к Prometheus как к источнику данных.
6.  Пользователи строят дашборды в Grafana и делают запросы на языке **PromQL**.

### Соответствие требованиям (Обоснование)

| Требование | Реализация |
| :--- | :--- |
| **Сбор метрик со всех хостов** | Запуск **Node Exporter** на каждом физическом или виртуальном хосте. |
| **Метрики ресурсов хостов (CPU, RAM, HDD, Network)** | Node Exporter собирает все эти метрики по умолчанию. |
| **Метрики потребляемых ресурсов для каждого сервиса** | **cAdvisor** (часто идет в комплекте с kubelet в Kubernetes) или Docker-демон. Отдает метрики потребления контейнеров, которые забирает Prometheus. |
| **Метрики, специфичные для каждого сервиса** | **Клиентские библиотеки Prometheus** (для Python, Java, Go и др.). Разработчик добавляет в код несколько строк, чтобы экспортировать метрики количества заказов, времени ответа БД и т.д. |
| **Пользовательский интерфейс с запросами и агрегацией** | **Prometheus Web UI** (базовый) и **Grafana Explore** (продвинутый). Используется язык запросов **PromQL**, который позволяет делать сложные агрегации (`sum by`, `rate`, `histogram_quantile`). |
| **Пользовательский интерфейс с настраиваемыми панелями (дашбордами)** | **Grafana**. Позволяет создавать полностью кастомизируемые дашборды: графики, таблицы, статус-панели. Поддерживается управление версиями дашбордов и общий доступ по ссылкам. |
